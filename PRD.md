## 1. Контекст

Компания использует цифровую платформу (портал), на которой проходит
полный цикл отбора и экспертизы НИОКР-проектов: от подачи заявителем
технологической задачи до получения решений и принятия итогового
решения по проекту.

Проблема: часть поступающих технологических задач фактически дублирует
уже существующие решения — запатентованные разработки или опубликованные
научные результаты. Сейчас это не проверяется системно, что создаёт риск
финансирования проектов, решение для которых уже существует на рынке
или в научной литературе.

## 2. Проблема и цель продукта

**Проблема:** эксперты не имеют инструмента для быстрой и системной
проверки новизны поступающей технологической задачи против внешних баз
патентов и научных публикаций. Проверка либо не делается вообще, либо
делается вручную, без единого стандарта, и занимает много времени.

**Цель:** дать порталу и экспертам инструмент, который на этапе подачи
заявки автоматически находит потенциально дублирующие документы во
внешних базах и присваивает заявке уровень риска дублирования, а также
позволяет эксперту дособрать похожие документы вручную при необходимости
более глубокой проверки.

**Не цель:** инструмент не принимает решение "отклонить/принять" —
решение всегда остаётся за экспертной комиссией. Сервис — это
вспомогательный сигнал, а не автоматический фильтр.

## 3. Метрики успеха

| Метрика | Как измеряем | Целевое значение (гипотеза, требует калибровки) |
|---|---|---|
| Доля заявок с автоматическим отчётом о похожести | заявки с отчётом / все поданные заявки | 100% после интеграции |
| Точность (precision) находок с высоким риском | доля high-risk находок, подтверждённых экспертом как реальный дубль | ≥ 60% на старте, рост по мере калибровки порогов |
| Полнота (recall) на известных дублях | доля исторически известных дублей, которые сервис бы нашёл (ретроспективный тест) | ≥ 70% |
| Время эксперта на проверку новизны одной заявки | до/после внедрения | сокращение не менее чем в 2 раза |
| Доля источников, реально доступных сервису | подключённые API / 4 целевых источника | рост от 1/4 (старт) до 4/4 (после интеграций) |

Precision/recall на старте нельзя гарантировать — это гипотезы, которые
уточняются в первые недели эксплуатации вместе с экспертами (см. раздел
9, "Калибровка").

## 4. Пользователи и сценарии использования

**Заявитель** — не является прямым пользователем сервиса, но косвенно
получает пользу: явные дубли отсекаются раньше, заявитель быстрее
получает обратную связь.

**Эксперт / member экспертной комиссии** — основной пользователь:
- получает автоматический отчёт о похожести вместе с заявкой на
  рассмотрение;
- может запустить ручной поиск с изменённой формулировкой запроса,
  если хочет проверить гипотезу о дублировании глубже, чем показал
  автоматический прогон.

**Администратор портала** — настраивает пороги риска, подключает/меняет
источники данных, следит за доступностью коннекторов.

### Сценарий 1 — автоматическая проверка при подаче
1. Заявитель подаёт технологическую задачу через портал.
2. Портал вызывает `POST /check` сервиса с названием и описанием задачи.
3. Сервис опрашивает подключённые источники, ранжирует найденные
   документы по похожести, возвращает отчёт с уровнем риска.
4. Отчёт прикрепляется к карточке заявки и виден эксперту на этапе
   экспертизы.

### Сценарий 2 — ручной поиск экспертом
1. Эксперт видит автоматический отчёт, у него есть сомнение или он хочет
   проверить другую формулировку задачи.
2. Эксперт через интерфейс (бот или форма в портале) отправляет
   уточнённый запрос в `POST /search`.
3. Получает тот же формат отчёта, может открыть найденные документы по
   ссылкам.

## 5. Функциональные требования

| # | Требование | Приоритет |
|---|---|---|
| F1 | Сервис принимает текст техзадания (название + описание) и возвращает список похожих документов с оценкой похожести | must |
| F2 | Каждой находке присваивается уровень риска: high / medium / low, пороги настраиваемые | must |
| F3 | Сервис опрашивает несколько источников параллельно и объединяет результаты в единый ранжированный список | must |
| F4 | Если источник недоступен (нет ключа, нет договора, сбой) — сервис не падает, а помечает источник как недоступный в отчёте | must |
| F5 | Отчёт содержит: название и ID найденного документа, источник, ссылку, сниппет, числовой score | must |
| F6 | Отдельный эндпоинт для автоматической проверки при подаче и для ручного поиска экспертом | should |
| F7 | Логирование решений эксперта (подтвердил / отклонил находку как реальный дубль) для последующей калибровки порогов и (в перспективе) дообучения ранжирования | should |
| F8 | Административная настройка порогов риска без изменения кода | could |
| F9 | Уведомление администратора, если ключевой источник (например, WIPO) стал недоступен | could |

## 6. Нефункциональные требования

- **Отказоустойчивость к внешним API:** недоступность одного источника
  не должна блокировать выдачу отчёта по остальным (см. F4).
- **Время ответа:** автоматическая проверка при подаче не должна
  ощутимо задерживать сценарий подачи заявки — целевой ориентир
  до нескольких секунд на источник, с возможностью асинхронного
  дозаполнения отчёта, если внешний API медленный.
- **Прозрачность отчёта:** эксперт должен видеть, какие источники
  реально были опрошены, а какие недоступны — риск ложного чувства
  полноты проверки хуже, чем явное "эта база не проверена".
- **Язык:** корректная обработка русскоязычных формулировок техзаданий
  как основной сценарий (не только английский).
- **Конфиденциальность:** тексты техзаданий могут содержать
  чувствительную информацию о заявителе — при выборе внешних API и
  условий передачи данных это нужно учитывать (см. риски, раздел 8).

## 7. Архитектура (высокий уровень)

```
Портал ──POST /check──▶ Novelty Checker API ──▶ Коннекторы источников (параллельно)
                              │                        │
                              ▼                        ▼
                     Similarity Engine        [патенты / публикации]
                              │
                              ▼
                       Отчёт о похожести
                              │
                              ▼
                 Карточка заявки в портале / ответ эксперту
```

Ключевое архитектурное решение: единый интерфейс коннектора
(источник → поиск → унифицированный формат документа), чтобы подключение
нового источника или замена одного источника на другой не требовали
изменений в остальной системе. Рабочий прототип с этой архитектурой и
кодом коннекторов передан отдельно (см. репозиторий `duplicate_checker`).

## 8. Источники данных и ограничения доступа

Ключевой риск проекта не технический, а организационный — доступ к
внешним базам:

| Источник | Готовность API | Что нужно сделать |
|---|---|---|
| Semantic Scholar | Доступен сейчас, без ключа | Готов к использованию как есть |
| WIPO PATENTSCOPE | Есть REST API | Регистрация разработчика на WIPO, получение ключа |
| Scopus | Есть API | Институциональная подписка + ключ Elsevier Developer Portal |
| Роспатент / ФИПС | Нет полноценного публичного API | Переговоры с ФИПС о канале доступа к данным, либо коммерческий агрегатор |
| eLibrary / РИНЦ | Публичного API нет вообще | Отдельный договор с НЭБ на выгрузку данных |

**Рекомендация:** не блокировать запуск ожиданием всех четырёх
источников. Стартовать с Semantic Scholar (и опционально CrossRef,
тоже бесплатный) для научных публикаций, параллельно вести переговоры
по WIPO (самый быстрый путь из патентных источников) и по остальным.

## 9. Этапы внедрения

**Этап 0 — прототип (готов).** Архитектура, рабочий коннектор к
Semantic Scholar, тестовый коннектор для демонстрации пайплайна на
патентных данных, REST API с двумя эндпоинтами.

**Этап 1 — пилот на одном источнике.** Подключение к порталу через
`/check`, реальные данные Semantic Scholar, TF-IDF как движок сравнения.
Сбор обратной связи от экспертов на реальных заявках.

**Этап 2 — калибровка.** На основе решений экспертов (F7) откалибровать
пороги high/medium/low, оценить реальные precision/recall, при
необходимости перейти с TF-IDF на эмбеддинг-модель
(`multilingual-e5-base` или аналог) для лучшего улавливания смысловых
дублей.

**Этап 3 — расширение источников.** По мере получения доступа —
подключение WIPO, затем Scopus/Роспатент/eLibrary по мере готовности
договорённостей.

**Этап 4 — интерфейс эксперта.** Отдельный бот или встроенная форма
поиска в портале поверх `/search`, если пилот покажет, что экспертам
нужен ручной дозапрос чаще, чем ожидалось.

## 10. Риски и допущения

- **Риск:** ключевые внешние базы (ФИПС, eLibrary) могут оставаться
  недоступны продолжительное время — сервис должен явно показывать
  неполноту проверки (F4), а не создавать ложное чувство полной
  проверки.
- **Риск:** baseline-движок сравнения (TF-IDF) может давать
  false negative на смысловых дублях, сформулированных другими словами —
  заложен переход на эмбеддинги как следующий шаг (см. Этап 2).
- **Риск:** пороги риска (high/medium/low) — гипотеза, не калиброванная
  на реальных данных организации; итоговые значения могут существенно
  отличаться от значений в прототипе (0.75 / 0.50).
- **Допущение:** эксперты готовы тратить время на разметку решений
  (F7) — без этого калибровка и последующее улучшение качества
  затруднены.
- **Допущение:** передача текста техзадания во внешние API (Semantic
  Scholar и др.) допустима с точки зрения конфиденциальности — требует
  подтверждения от юридической/безопасности службы организации.

## 11. Открытые вопросы

1. Кто в организации будет вести переговоры с ФИПС и НЭБ о доступе к
   данным — и в какие сроки?
2. Есть ли у организации уже действующая подписка на Scopus, которую
   можно переиспользовать?
3. Кто будет проводить разметку (F7) — сами эксперты в рамках обычной
   работы, или нужен отдельный процесс?
4. Нужен ли асинхронный режим проверки (если внешний источник отвечает
   долго), или синхронный ответ в рамках подачи заявки достаточен?
5. Где размещается сервис — на инфраструктуре, где уже стоит портал,
   или отдельно, с интеграцией по API?
